Plan-It Trails was conceived, built, and deployed entirely between Sep. 14 - Oct. 9 2020 as my project for Insight Fellows Data Science program (fall session, 2020C-IDX). This was a whirlwind of an experience and I hope you enjoy using this app. If you find a bug feel free to submit it as an issue on Github, or drop me an e-mail at aemerick11 - AT - gmail.com. But most importantly, if you use this app to generate a trail route and enjoy it, do let me know!

The details: PlanIt-Trails leverages the OpenStreetMaps database to enable trial optimization world-wide. To generate the optimized routes, PlanIt-Trails uses the open-source Python packages OSMNx to fetch the OpenStreetMap data. OSMNx uses Networkx, to build a graph object defining edges and nodes of the graph as the trail segments and their intersections. Plan-It layers in a digital elevation model from NASA's Shuttle Radar Topography Mission (parsed via: srtm) in order to define altitude, obtain more accurate distances, compute elevation gain, elevation loss, and min, max, and average grade along each segment.

The algorithm uses careful weighting of the trail edges to incentivise routing along (or de-incentivise travel along) specific trail segments as the route is built. The algorithm begins by choosing a "next-node" to travel to by obtaining the list of nodes within a paramaterized distance from the current node, and selecting one that does not require violating the provided constraints to travel to. Then a shortest-path Dijkstra algorithm (implemented in Networkx) is used to compute the weighted shortest path to that node. The result is added to the route build so far, and this process is iterated upon until the route begins to reach near its distance contraint. At this point it is guided back home / to the end point. This dual iteration process, combined with a careful, dynamic weighting of the feature space are what makes Plan-It work as well as it does.

Known Issues and Future Work: You can learn A LOT about the quality of a project by its ability to openly recognize its faults and limitations, and plans for future work. There are a few known issues that I would like to improve upon and new work I'd like to do. This includes:

  1. Accurate time-to-completion estimation. Perhaps what I'm most excited to work on, and my initial Insight project idea. In short, training a linear ML model on one's Strava activity history to make accurate predictions for route time-to-completion, improving dramatically over Strava's current calculator and allowing users to constrain on length of activity.
  2. Additional datasets. Building off of prior Insight fellows projects, it would be extremely interesting to layer in trail data from TrailRunningProject / HikingProject, using the user reviews and ratings from those trails to add in a "quality" feature to each segment to add the ability to try and force Plan-It to stay on generally more highly rated or popular paths (or actively avoid the most-well-travelled).
  3. Locally storing OSM. OpenStreetMap data is currently fetched for a given area each time, which makes things slow. While I use some caching to speed this up, it is very basic. With more time and resources I'd like to download the entire OSM database locally and store a fully engineered version on the server, allowing for more rapid and arbitrary graph construction.
  4. UX. Let's face it, I'm not a front-end developer so the UI needs a ton of work. But a really cool feature that would make this a more powerful tool is the ability to take the generated route and then arbitrarily edit / cut / paste the generated route to smooth over any weird quirks or direct it to cool places that it may have missed. Right now if you want to edit a route you build in any other existing app you have to delete segments from the end to where you want to edit, make the edit, then rebuild. Annoying!.
  5. Currecting inaccurate and missing data The OpenStreetMap database does not always provide reliable trail junctions. This limits the ability to build routes in some cases. It is challenging to come up with an algorithm to reliably stitch these together while also leaving ones that truly are separated separate. This is a mostly solved problem if I had access to a more reliable database like the background trail maps that AllTrails uses. Related to this, I'd like to also allow for travel over roads to connect trails which may be separated only by a road or a parking lot. Though doing this well again provides some unique challenges.

Hitting the Trails

By using PlanIt-Trails, the user accepts all liability for anything that may occur during an activity inspired by or following the generated routes. PlanIt-Trails does not guarantee that the underlying trail data is up to date or accurate, nor that the recommended trails are safe to travel upon. Always check local land-use restrictions and potential closures before hitting the trails. Be smart! Be safe! Leave no trace! And have fun!
